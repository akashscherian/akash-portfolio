Security project · 2024

Home Lab SOC — Wazuh SIEM/XDR

A self-hosted detection lab built to learn the SOC analyst workflow end-to-end — agent deployment, log ingestion, rule tuning, and attack simulation — with every alert mapped to MITRE ATT&CK, PCI DSS, and GDPR.

Wazuh SIEM/XDR OpenSearch Docker MITRE ATT&CK Nmap Linux/macOS

Why I built this

Most cybersecurity learning material talks about SIEMs abstractly. I wanted to actually feel the workflow a SOC analyst lives in every day: an alert fires, you triage it, you look at the raw log, you check the rule, you decide whether it's a true positive, and either escalate or tune. The only way to internalise that loop is to run it on your own machine.

Wazuh was the obvious choice — open source, full SIEM + XDR + FIM + SCA in one stack, with native MITRE ATT&CK mapping. The challenge: get it running on Apple Silicon (ARM64), which isn't the platform Wazuh's official documentation assumes.

Architecture

Three Wazuh services plus a monitored endpoint, all containerised and orchestrated with Docker Compose.

What I configured

1 — Wazuh stack on Docker (ARM64)

The default Wazuh deployment assumes x86_64. On Apple Silicon, several Indexer images would crash on boot. The fix was using the official multi-arch images plus pinning a compatible Wazuh version (4.7.x), then bumping container memory limits because the Indexer's JVM is hungry:

docker compose -f docker-compose.yml up -d
# Verified all three services healthy:
docker compose ps

2 — File Integrity Monitoring (FIM)

Configured FIM on the agent's ossec.conf to watch sensitive directories in real time. Real-time mode catches changes immediately rather than on a scheduled scan, which matters for catching attackers who modify a binary and clean up quickly.

<syscheck>
  <directories check_all="yes" realtime="yes">/etc</directories>
  <directories check_all="yes" realtime="yes">/usr/bin</directories>
  <directories check_all="yes" realtime="yes">/usr/sbin</directories>
</syscheck>

3 — Rootcheck & Security Configuration Assessment (SCA)

Rootcheck runs anomaly checks for rootkit signatures, hidden processes, and known trojan patterns. SCA runs CIS Benchmark checks on the host and reports compliance gaps. Both feed into the same alert pipeline.

Attack simulations

A SIEM with no test traffic is just an expensive logger. I ran two simulations to validate detection.

Simulation 1 — trojaned binary

  1. Took a checksum baseline of /usr/bin/ls with the agent running.
  2. Replaced the binary with a benign altered copy (added a comment, kept the file executable).
  3. FIM flagged the modification within seconds — alert level 7, mapped to MITRE T1565.001 (Stored Data Manipulation).
What I learned: the alert isn't useful on its own — what makes it useful is the rule's mapping to PCI DSS Requirement 11.5 (deploy file-integrity-monitoring software) and the analyst's ability to pivot from the alert into the raw syscheck event with the old/new hash side by side.

Simulation 2 — unauthorised port scan

  1. Ran nmap -sS -p 1-1000 <target-ip> against the monitored host from another machine on the LAN.
  2. Multiple connection-attempt log lines aggregated into a single grouped alert via the manager's correlation rule.
  3. Alert auto-tagged with MITRE T1046 (Network Service Scanning) and surfaced in the Threat Hunting dashboard.

Tuning & false-positive reduction

Out of the box, the lab fired ~30 alerts/hour from routine background activity — package manager checks, scheduled jobs, log rotation. That's noise that buries real signals. I tuned in three passes:

  • Local rule overrides: wrote custom rules in local_rules.xml to suppress known-benign events (LaunchDaemon refreshes, Homebrew updates) without touching the upstream ruleset.
  • FIM exclusions: excluded high-churn directories (/var/log, /tmp) from real-time monitoring — they generate FIM noise without security value.
  • Alert-level threshold: set the dashboard's default filter to level ≥ 5, hiding informational events from the triage view while still indexing them for hunting.

After tuning, alert volume dropped roughly 80% — and the alerts that remained were ones I could actually act on.

What this taught me

  • The mechanics of SIEM agent ↔ manager ↔ indexer ↔ dashboard, and what breaks when you skip a step.
  • How MITRE ATT&CK and compliance frameworks aren't separate things from detection — they're the metadata that makes alerts useful to a business.
  • Detection engineering is more about tuning out noise than writing clever rules. The starter ruleset is usually good; making it useful is the job.
  • Reproducible deployment matters. Docker Compose means I can blow the lab away and rebuild it in 5 minutes — which I did several times while learning.

What I'd add next

  • A second agent on a Linux VM to test cross-platform rule consistency.
  • Wazuh's Active Response feature to auto-block IPs after N failed SSH attempts.
  • Send alerts to a Slack webhook for mobile triage — a realistic SOC pattern.
  • Replay a public APT dataset (Atomic Red Team) and measure detection coverage by ATT&CK technique.
